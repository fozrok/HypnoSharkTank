# HypnoSharkTank: App Changes Specification
## "Shark Tank Feeding Frenzy" Feature

> **Document Purpose:** This specification describes all changes required to integrate the Shark evaluation panel into the existing HypnoSharkTank app flow. It covers the renamed feature, new UI interactions, selection mechanics, advice generation, approval workflow, and how approved Shark advice becomes persistent context for all downstream features.

---

## 1. Summary of Changes

The existing 7-feature prompt chain is modified as follows:

| # | Previous Name | New Name | Changed? |
|---|---|---|---|
| 1 | Niche Opportunity Finder | Niche Opportunity Finder | No change |
| 2 | Business Model Designer | Business Model Designer | No change |
| 3 | Market Validation Sniper | **Shark Tank Feeding Frenzy** | **Renamed and redesigned** |
| 4 | Revenue & Growth Projection | Revenue & Growth Projection | **Context updated** (receives Shark advice) |
| 5 | Go-to-Market Messaging | Go-to-Market Messaging | **Context updated** (receives Shark advice) |
| 6 | 30-Day Launch Sprint | 30-Day Launch Sprint | **Context updated** (receives Shark advice) |
| 7 | Brutal Reality Check | Brutal Reality Check | **Context updated** (receives Shark advice) |

**Key principle:** Features 1-2 remain untouched. Feature 3 becomes a multi-step interactive experience (the Feeding Frenzy). Features 4-7 are unchanged in structure but receive a new context payload: the collection of approved Shark advice that shapes all subsequent outputs.

---

## 2. Feature 3: Shark Tank Feeding Frenzy

### 2.1 Overview

The Shark Tank Feeding Frenzy replaces the previous Market Validation Sniper. It is no longer a single prompt that returns a go/pivot/kill verdict. Instead, it is a multi-step interactive experience where 7 Sharks each evaluate the business idea independently, offer their verdict, and (if investing) present actionable help the user can select and approve.

The Feeding Frenzy produces two outputs:
1. **A panel verdict** (replaces the old go/pivot/kill): determined by how many Sharks invest.
2. **A collection of approved Shark advice** (new): specific strategic guidance from each invested Shark that the user has reviewed and approved.

### 2.2 Entry Conditions

The user can only enter the Shark Tank Feeding Frenzy when:
- Feature 1 (Niche Opportunity Finder) is complete and a niche has been selected
- Feature 2 (Business Model Designer) is complete with at minimum: offer name, target client, pricing, delivery model, and positioning statement

The app validates that these fields exist before allowing the user to proceed. If any are missing, the app displays a clear message identifying what needs to be completed and a link back to the relevant feature.

## 3. Feeding Frenzy: Step-by-Step Flow

### Step 1: The Pitch (Automatic)

**What happens:** The app assembles the user's business idea into a structured "pitch" using data from Features 1-2.

**Pitch structure (assembled automatically):**
```
BUSINESS IDEA PITCH
===================
Niche: [from Feature 1]
Offer Name: [from Feature 2]
Target Client: [from Feature 2]
Transformation Promise: [from Feature 2]
Delivery Model: [from Feature 2]
Pricing: [from Feature 2]
Scalability Roadmap Summary: [from Feature 2]
Positioning Statement: [from Feature 2]
Practitioner Background: [from Feature 1 intake]
```

**UI:** The pitch is displayed as a clean summary card at the top of the Feeding Frenzy screen, so the user can review what the Sharks will see. An "Edit Pitch" button allows the user to return to Feature 2 to make changes before proceeding.

**Action:** The user clicks "Release the Sharks" to begin the evaluation.

---

### Step 2: Shark Evaluations (Sequential Reveal)

**What happens:** All 7 Sharks evaluate the pitch simultaneously (7 parallel API calls using the Shark persona prompts from the Shark Personas system file). However, the results are revealed to the user one at a time with a brief dramatic pause between each, building anticipation.

**Per-Shark evaluation output (generated by AI):**
```json

  ],
  "critical_question": "If this program delivers the result you promise, what is that result worth to your client in dollars, career longevity, or quality of life over the next 12 months?",

    {
```

**Reveal sequence:**
1. Each Shark card appears one at a time (0.8-1.2 second delay between reveals)
2. The verdict badge animates in after a brief pause (adds tension)
3. Invested Sharks show their 3 help buttons immediately
4. Passed Sharks show their "What Would Change My Mind" statement (generated in place of help options)
5. Conditional Invest Sharks show their condition prominently, plus 3 help buttons (greyed out until the user acknowledges the condition)

**Reveal order:** Randomized per session to prevent predictability. However, the app should avoid revealing all passes first or all invests first. A mixed sequence creates better engagement.

### Step 3: Help Selection (User Interaction)

**What happens:** After all 7 Sharks have been revealed, the user enters the selection phase. For each Shark that invested (or conditionally invested with condition acknowledged), the user must select exactly 1 of the 3 help buttons.

**Rules:**
- Each invested Shark presents exactly 3 help buttons
- The user must select exactly 1 button per invested Shark
- Selecting a button highlights it and dims the other 2 for that Shark
- The user can change their selection before confirming (clicking a different button deselects the previous one)
- Passed Sharks have no buttons and are visually separated (dimmed, moved to a "Sharks Who Passed" section below)
- The user cannot proceed until they have made a selection for every invested Shark

**UI state during selection:**

```
INVESTED SHARKS (Select 1 help option per Shark)
=================================================

ALEX HORMOZI - The Offer Architect [INVEST]
  ( ) Restructure My Offer
  (•) Fix My Pricing              <-- selected
  ( ) Add Recurring Revenue

SETH GODIN - The Positioning Strategist [INVEST]
  (•) Refine My Positioning        <-- selected
  ( ) Define My Smallest Audience
  ( ) Build My Permission Asset

STEVE JOBS - The Experience Perfectionist [INVEST]
  ( ) Simplify My Offer
  ( ) Design My Onboarding Moment
  ( ) [not yet selected]           <-- user must choose

... (remaining invested Sharks)

=================================================
SHARKS WHO PASSED
=================================================
SARA BLAKELY - The Scrappy Launcher [PASS]
  "What would change my mind: ..."
  [dimmed, no interaction]

=================================================
              [ Confirm Selections ]
              (disabled until all invested Sharks have a selection)
```

**Confirmation:** When the user clicks "Confirm Selections," a brief confirmation modal appears: "You've chosen [X] areas of help from [Y] Sharks. Ready to hear their advice?" with "Let's Go" and "Change My Mind" buttons.

### Step 4: Advice Generation (AI-Generated Per Selection)

**What happens:** For each selected help button, the app generates detailed advice from that Shark's perspective, within the context of the user's specific business idea.

**Prompt construction per advice generation:**
```
System: [Full Shark persona from Shark Personas file]
Context: [User's pitch data from Features 1-2]
Selected help: [The specific button label and description the user chose]

Generate advice as this Shark. The advice should:
- Be 300-500 words
- Be specific to the user's business idea (reference their niche, offer, pricing, client by name)
- Provide actionable, implementable guidance (not theory)
- Include at least one concrete example or specific recommendation
- End with a clear "next step" the user can take immediately
- Stay in character as this Shark
```

**UI:** Each Shark's advice is displayed as an expandable card. The advice appears with a brief typing animation (as if the Shark is writing it), reinforcing the persona experience.

```
+----------------------------------------------------------+
|  ALEX HORMOZI'S ADVICE: Fix My Pricing                    |
|  -------------------------------------------------------- |
|                                                           |
|  [300-500 words of specific, actionable pricing advice    |
|   tailored to the user's exact business idea, referencing |
|   their niche, offer name, target client, and current     |
|   pricing. Ends with a clear next step.]                  |
|                                                           |
|  -------------------------------------------------------- |
|                                                           |
|  Do you accept this advice?                               |
|                                                           |
|  [ AGREE - Apply to my plan ]    [ DISAGREE - Skip this ] |
|                                                           |
+----------------------------------------------------------+
```

### Step 5: Agree / Disagree (User Approval)

**What happens:** For each piece of Shark advice, the user must indicate whether they agree (accept) or disagree (reject). This is a critical step because only agreed-upon advice becomes part of the persistent context for downstream features.

**Rules:**
- Each advice card has exactly 2 buttons: "Agree" and "Disagree"
- Agreeing adds the advice to the "Approved Shark Advice" collection
- Disagreeing removes it. The Shark's general evaluation (score, observations) remains visible for reference, but their specific advice does not flow into downstream features
- The user can revisit and change their agree/disagree decisions before leaving the Feeding Frenzy
- A running tally shows: "You've approved X of Y Shark recommendations"

**UI after all decisions are made:**

```
SHARK TANK FEEDING FRENZY - SUMMARY
====================================

Panel Result: 5 of 7 Sharks Invested
Approved Advice: 4 of 5 (you disagreed with 1)

APPROVED ADVICE (will shape your plan going forward):
  ✓ Hormozi: Fix My Pricing
  ✓ Godin: Refine My Positioning  
  ✓ Jobs: Design My Onboarding Moment
  ✓ Da Vinci: Cross-Disciplinary Innovation

DECLINED ADVICE:
  ✗ Naval: Automation strategy (you disagreed)

SHARKS WHO PASSED:
  — Blakely (would need a faster launch path)
  — Forleo (would need a clearer content strategy)

              [ Continue to Revenue Projections >> ]
```

## 4. Approved Shark Advice: Persistent Context

### 4.1 What Gets Stored

When the user completes the Feeding Frenzy, the following data is stored as persistent context:

```json

  "feeding_frenzy_result": {
    "total_sharks": 7,
    "invested": 5,
    "passed": 2,
    "approved_advice": [
      {
        "shark": "Alex Hormozi",
        "title": "The Offer Architect",
        "selected_help": "Fix My Pricing",
        "advice_text": "[full 300-500 word advice]",
        "status": "approved",
        "key_recommendation": "Increase pricing to $1,997 with 3-pay option at $747. Add a guarantee."
      },
      {
        "shark": "Seth Godin",
        "title": "The Positioning Strategist",
        "selected_help": "Refine My Positioning",
        "advice_text": "[full 300-500 word advice]",
        "status": "approved",
        "key_recommendation": "Narrow target to mid-career nurses who dread going to work. Lead with their language."
      }
      // ... additional approved advice
    ],
    "declined_advice": [
      {
        "shark": "Naval Ravikant",
        "selected_help": "Automate My Delivery",
        "status": "declined"
      }
    ],
    "passed_sharks": [
      {
        "shark": "Sara Blakely",
        "reason": "Cannot launch within 7 days as designed"
      }
    ]
  }

```

Every prompt for Features 4-7 now receives an additional context block injected before the feature's primary prompt:

```
APPROVED ADVICE:

   Key recommendation: [key_recommendation text]
   Full context: [abbreviated advice_text, ~100 words]

2. Seth Godin (The Positioning Strategist) - "Refine My Positioning"
   Key recommendation: [key_recommendation text]
   Full context: [abbreviated advice_text, ~100 words]

   Key recommendation: [key_recommendation text]
   Full context: [abbreviated advice_text, ~100 words]

```

### 4.3 Feature-by-Feature Integration

**Feature 4: Revenue & Growth Projection**
- Uses approved pricing advice (e.g., Hormozi's $1,997 recommendation) as the default pricing assumption in the model
- If a Shark recommended a different revenue model (e.g., subscription, tiered), the projection should model that structure
- References specific Shark advice in the assumptions section: "Based on your approved pricing strategy (Hormozi), the model uses $1,997 per client with a 3-pay option."

**Feature 5: Go-to-Market Messaging**
- Uses approved positioning advice (e.g., Godin's narrowed audience) to shape all hook writing and outreach sequences
- If Jobs advised on simplifying the offer description, the messaging should reflect that simpler framing
- The trust-language cheat sheet should align with the approved Shark direction (e.g., if Godin's positioning was approved, the language should target the specific tribe he defined)

**Feature 6: 30-Day Launch Sprint**
- Daily actions should reflect approved advice. For example:
  - If Hormozi's pricing was approved, Week 1 includes "Update all pricing to $1,997 with 3-pay option"
  - If Jobs' onboarding redesign was approved, Week 2 includes "Implement new onboarding sequence"
  - If Da Vinci's innovation was approved, the sprint includes time to develop the innovative element
- The sprint should not include actions that contradict declined advice (e.g., if the user declined Naval's automation advice, don't fill the sprint with automation tasks)

**Feature 7: Brutal Reality Check**
- The reality check auditor should evaluate the plan in light of approved Shark advice
- It should flag potential conflicts between different Sharks' approved advice (e.g., "Hormozi told you to price at $1,997 and Godin told you to narrow your audience to mid-career nurses. Be aware that the combination of premium pricing + narrow audience means your total addressable market is smaller. Here's whether the math still works...")
- It should also note the concerns raised by Sharks who passed, as these represent potential blind spots the approved advice may not address

## 5. Conditional Invest Handling

When a Shark delivers a "Conditional Invest" verdict, the flow is slightly different:

1. The Shark's card displays the condition prominently:
```
+----------------------------------------------------------+
|  NAVAL RAVIKANT - The Leverage Engineer                   |
|                                                           |
|  Score: 6/10               VERDICT: CONDITIONAL INVEST    |
|  ██████░░░░                [amber badge]                  |
|                                                           |
|  CONDITION:                                               |
|  "I'll invest IF you commit to building at least one      |
|   revenue stream that doesn't require your live presence. |
|   Without that, this is a practice, not a business."      |
|                                                           |
|  [ I Accept This Condition ]   [ I Don't Accept ]         |
|                                                           |
+----------------------------------------------------------+
```

2. If the user accepts the condition:
   - The Shark's 3 help buttons become active
   - The user selects 1, receives advice, and agrees/disagrees as normal
   - The accepted condition is stored as additional context alongside any approved advice

3. If the user does not accept the condition:
   - The Shark is treated as a Pass
   - Their card dims and moves to the "Sharks Who Passed" section
   - No help buttons are available

## 6. Edge Cases

### What if no Sharks invest?

If 0 Sharks invest (all pass or all conditions are rejected):

- Display a "No Deal" screen with a clear but encouraging message
- Show each Shark's "What Would Change My Mind" statement
- Provide two options:
  1. "Revise My Business Model" (returns to Feature 2 with Shark feedback visible as reference)
  2. "Try a Different Niche" (returns to Feature 1)
- Do not allow the user to proceed to Feature 4 without at least 1 Shark investment

### What if only 1-2 Sharks invest?

- Allow the user to proceed, but display a warning: "Only [X] of 7 Sharks invested. Your plan will have limited strategic input. Consider revising your business model to address the concerns raised by Sharks who passed."
- The downstream context will contain fewer approved advice items, which is fine. Features 4-7 should still function with minimal context.

### What if the user agrees with all advice?

- Normal flow. All approved advice enters the context. Downstream features have maximum strategic input.

### What if the user disagrees with all advice?

- Allow it. The Feeding Frenzy result shows "X Sharks invested, 0 advice approved."
- Downstream features receive the Feeding Frenzy scores and verdicts as context (so they know the idea was evaluated), but no specific Shark advice shapes the outputs.
- The app should note: "You chose not to adopt any Shark recommendations. Your plan will proceed based on your original business model."

### Rerunning the Feeding Frenzy

- The user can rerun the Feeding Frenzy at any time after modifying their Feature 1 or Feature 2 outputs
- Rerunning clears all previous Shark evaluations, selections, and approved advice
- Downstream features are flagged as "stale" if they were previously generated with old Shark advice

## 7. Data Model

### New Entities

```
SharkEvaluation {
  id: UUID

  id: UUID

  invested_count: Integer

```

## 8. UI/UX Notes

### Visual Design (Applicable to All 3 Design Options)

- **Shark avatars:** Each Shark should have a distinctive, recognizable avatar. Use stylized illustrated portraits rather than photographs (avoids likeness rights issues and creates a cohesive visual language). Each avatar uses a unique accent color.
- **Verdict animations:** Invest = green glow pulse. Conditional = amber pulse. Pass = card dims with a subtle fade.
- **Help buttons:** Styled as prominent, tappable cards (not small text links). Each button has a label (bold, max 30 characters) and a one-line description below.
- **Advice cards:** Use the Shark's accent color as a left border stripe. The advice text should feel personal, not clinical.
- **Agree/Disagree buttons:** Large, clear, no ambiguity. "Agree" is styled as the primary action (filled button). "Disagree" is styled as secondary (outline button). Neither should feel punitive.

### Gamification Elements

- **Shark count badge:** A persistent indicator during the Feeding Frenzy showing "X / 7 Sharks Invested" that updates as each Shark reveals their verdict.
- **Sound effects (optional):** A subtle "splash" or "chomp" sound when a Shark invests. A quieter "retreat" sound when they pass. Keep it playful, not cheesy.
- **Completion celebration:** When the user finishes the Feeding Frenzy with 5+ Sharks invested, a brief celebration animation plays (confetti, the invested Shark avatars assembling into a "team" graphic, etc.).
- **Titles based on investment count:**
  - 7 Sharks: "Feeding Frenzy Champion"
  - 5-6 Sharks: "Strong Contender"
  - 3-4 Sharks: "In the Game"
  - 1-2 Sharks: "Underdog"
  - 0 Sharks: "Back to the Drawing Board"

### Responsive Behavior

- **Desktop:** Shark cards display in a 2-column grid. Help buttons display side by side.
- **Tablet:** Shark cards display in a single column. Help buttons display side by side.
- **Mobile:** Shark cards are full-width, swipeable. Help buttons stack vertically. The sequential reveal works especially well on mobile (swipe through Sharks one at a time).

## 9. Prompt Engineering Notes

### Generating the 3 Help Buttons

The help buttons must be:
- **Distinct:** Each button addresses a different aspect of the Shark's expertise. No overlap.
- **Specific to the idea:** The buttons should reference the user's actual business model, not generic options.
- **Actionable:** Each button label should start with a verb (Restructure, Fix, Add, Simplify, Design, Define, Build, Automate, Create, Refine, etc.)
- **Appropriate to the verdict:** If a Shark gives a high score (8-10), the buttons should be refinement-oriented. If the score is lower (5-7), the buttons should address more fundamental issues.

**Prompt instruction for generating help buttons:**
```
Generate exactly 3 help options. Each must:
- Address a different dimension of your expertise
- Be specific to this user's business idea (reference their niche, offer, or client)
- Have a button label of 4 words or fewer starting with a verb
- Have a description of 1 sentence (max 20 words)
- Not overlap with the other 2 options
- Represent genuinely different strategic directions (not just variations of the same advice)
```

### Generating Advice After Button Selection

**Prompt instruction for generating advice:**
```
The user selected your help option: "[button_label]"
Description: "[description]"

Generate your advice for this specific business idea. Your advice must:
- Be 300-500 words
- Reference the user's niche, offer name, pricing, target client, and delivery model by name
- Provide specific, implementable recommendations (exact numbers, exact steps, exact language)
- Include at least one concrete example showing before/after
- End with one clear next action the user can take within 24 hours
- Stay fully in character as [Shark name]
- Build on your original observations and critical question from the evaluation
- Do not repeat your evaluation observations verbatim; expand on them with new depth

Also generate a key_recommendation field: a single sentence (max 30 words) summarizing the core strategic shift you are advising. This will be used as context in downstream features.
```

### Context Injection for Features 4-7

**Token management:** The full advice text from all approved Sharks could be lengthy. To manage token usage:
- Inject the `key_recommendation` for each approved Shark (always, this is the priority)
- Inject an abbreviated version of the full advice (first 100 words) for up to 5 approved Sharks
- If there are more than 5 approved items, summarize the remaining ones as key recommendations only
- Always include the notes about declined advice and passed Sharks (brief, 1-2 sentences each)

**Priority order for context injection:**
1. Key recommendations from all approved Sharks (highest priority, always included)
2. Abbreviated advice text from approved Sharks (included if token budget allows)
3. Declined advice notes (brief, always included)
4. Passed Shark concerns (brief, always included)
5. Scores and verdicts from all 7 Sharks (included if token budget allows)
